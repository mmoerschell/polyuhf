\chapter{Introduction}
\label{ch:introduction}

Universal hash functions are a small but central component in many cryptographic constructions~\cite{carter1979universal}. They turn fixed-key algebraic operations into fast authenticators, for example in Carter-Wegman style message authentication codes and authenticated encryption schemes such as AES-GCM and ChaCha20-Poly1305. Their security is information-theoretic at the hash-family level, while their performance is often close to ordinary arithmetic on machine words.

The deployed example that motivates much of the recent literature is Poly1305. It evaluates a polynomial over the prime field \(2^{130}-5\) and is widely used as part of ChaCha20-Poly1305~\cite{bernstein2005poly1305aes}. Its performance depends on implementation choices such as the prime field, limb representation, multiplication algorithm, carry placement, and target vector ISA. Small changes in these choices can change both throughput and the security obtained from a fixed number of input bytes.

Recent work has begun to systematize this design space. Degabriele, Gilcher, Govinden, and Paterson collect implementation techniques for polynomial hashing over prime fields and propose alternatives to Poly1305~\cite{degabriele2024sok}. SPHGen turns part of this space into a program generator for simple polynomial hash functions over Crandall prime fields~\cite{pegolotti2026sphgen}. New multivariate polynomial designs combine lower-level and higher-level polynomial hash functions~\cite{gilcher2026newdesigns}. These developments suggest a compiler question: can we describe polynomial hash families at a higher level, while still generating efficient low-level code over a range of fields and instruction sets?

\section{Problem Statement}
\label{sec:problem-statement}

This thesis designs and implements a small domain-specific language (DSL) for polynomial universal hash functions. The language lets a designer describe algebraic structure without manually writing field arithmetic, vectorized loops, carry propagation, and test harnesses. The compiler lowers this description to C code for concrete prime fields and target platforms.

The useful abstraction boundary is not obvious. A language that exposes only low-level arithmetic parameters is easy to compile, but does not express newer polynomial designs. A language that hides all implementation choices risks preventing the optimizations that make prime-field hashing fast. This project explores a middle ground: expressive enough for lower-level designs from the multivariate-polynomial literature, but close enough to SPHGen's implementation model to support vectorization and specialized field arithmetic.

\section{Design Goals}
\label{sec:design-goals}

The compiler is guided by four design goals.

First, the source language should reflect the algebraic shape of polynomial hash functions, with direct constructs for sums, Horner-style evaluation, and left folds.

Second, the compiler should separate the mathematical description from machine-dependent choices. The same DSL module should compile for different Crandall prime fields, multiplication algorithms, and vector targets.

Third, the generated code should expose a stable hash-function API over byte buffers and a byte length, while internally mapping bytes to field elements according to the selected field.

Fourth, the implementation should come with practical validation tools: generated tests based on a high-precision C++ reference implementation, and handwritten tests where an external reference is available.

\section{Contributions}
\label{sec:contributions}

\begin{itemize}
    \item We design a DSL for polynomial universal hash functions over Crandall prime fields, with field elements, indices, buffers, helper functions, hash functions, arithmetic expressions, conditionals, and three reduction constructs.
    \item We define an intermediate representation (IR) for scalar and vectorized prime-field arithmetic, with explicit instructions, loops, conditionals, loads, stores, carry propagation, and full reductions.
    \item We implement a C backend for scalar code, ARM NEON, and AVX2. The backend emits field arithmetic for Crandall prime fields, including schoolbook multiplication, Karatsuba multiplication, dedicated squaring, load/store routines, carry propagation, and full reduction.
    \item We implement compiler lowering for the lower-level polynomial patterns that appear in the new-designs literature, including sums for MMH/SQH/NMH-style constructions, Horner-style evaluation for Poly1305-like functions, and left folds for iterative HKM-style recurrences.
    \item We generate correctness tests using Boost Multiprecision~\cite{boost}, and add manual validation tests for Poly1305 and HKM.
    \item We generate a benchmarking harness and conduct experiments to identify promising designs.
    \item We present a performance model for Apple silicon ARM NEON chips, deriving runtime estimates from the IR.
\end{itemize}

The code for this project is available at \url{https://github.com/mmoerschell/polyuhf/}.
